# 模型管理与 OpenAI 兼容多模型对话设计

## 背景

当前系统的对话链路是固定运行时：

1. Web 工作台发送消息到 `POST /api/sessions/{session_id}/messages`
2. API 服务调用 `messages_service.handle_user_message`
3. `run_agent(state, user_input)` 返回固定回复与动作

这套链路还不支持：

1. 平台级模型配置管理
2. 管理员控制模型启用状态
3. 工作台按消息选择模型
4. 基于 `base_url + api_key + model` 发起真实外部推理请求

本次设计要补齐这一条完整链路。

## 目标

本次实现需要满足以下目标：

1. 只有管理员可以管理模型配置。
2. 管理员通过页面维护模型的 `name`、`base_url`、`api_key`、`model`、`enabled`。
3. 管理权限通过环境变量 `ADMIN_EMAILS` 的邮箱白名单判定。
4. 普通用户在工作台中只能看到已启用模型。
5. 用户每次发送消息前都可以切换模型，本次发送按当前选择的模型生效。
6. 后端按 OpenAI 兼容接口使用对应的 `base_url + api_key + model` 发起真实对话请求。
7. 历史消息中保留本次使用的模型元数据，方便审计和回放。

## 非目标

本次不包含以下范围：

1. 不支持 Anthropic、Gemini 等非 OpenAI 兼容协议。
2. 不引入复杂 RBAC 或数据库级角色系统。
3. 不做模型可用性探测、测速或健康检查页面。
4. 不做模型调用自动降级或自动切换。
5. 不处理 API Key 加密存储，本期按需求采用明文存储和明文回显。

## 方案选择

### 方案 A：在现有架构上新增模型注册表与 OpenAI 兼容适配层

内容：

1. 新增平台级模型配置表。
2. 新增管理员模型管理页。
3. 工作台发送消息时附带 `model_config_id`。
4. 后端按模型配置请求 OpenAI 兼容接口。

优点：

1. 与当前需求完全一致。
2. 改动集中在现有消息链路周围，落地速度快。
3. 能最大化复用现有会话、SSE、Zustand 和消息存储结构。

缺点：

1. 当前只支持一种协议抽象。
2. 后续扩展到多协议提供商时需要再抽象一层。

### 方案 B：先做通用多提供商抽象层

优点是长期扩展更强，缺点是当前阶段明显过重，会拉高实现和测试复杂度。

### 结论

采用方案 A。

原因：

1. 它直接覆盖管理员配置、前台选择、后端真实调用这三条主需求。
2. 当前代码库的对话链路还比较薄，先做 OpenAI 兼容适配比提前做过度抽象更合适。

## 角色与权限设计

### 管理员判定

管理员身份不落数据库，按环境变量白名单判定：

1. 新增环境变量 `ADMIN_EMAILS`
2. 取值为邮箱列表
3. 当前登录用户邮箱命中白名单则视为管理员

### 权限规则

1. 管理员可以访问模型管理 API 和管理页面。
2. 普通用户不能访问管理页面和管理 API。
3. 普通用户可以访问“已启用模型列表”接口，用于工作台选择。
4. 普通用户可以在自己的会话中使用已启用模型发送消息。

## 数据模型设计

新增平台级模型配置表，建议命名为 `llm_model_configs`。

字段建议如下：

```text
id: string
name: string
base_url: string
api_key: string
model: string
enabled: boolean
created_at: datetime
updated_at: datetime
```

字段说明：

1. `id`
   前后端选择模型时使用的主键。
2. `name`
   管理页和工作台展示名，例如“DeepSeek V3 生产”。
3. `base_url`
   OpenAI 兼容接口根地址。
4. `api_key`
   本期按需求明文保存。
5. `model`
   外部请求使用的模型名。
6. `enabled`
   控制是否对工作台开放。

### 历史消息元数据

会话本身不绑定默认模型，因为模型选择是消息级行为。

因此模型使用信息应写入消息 `meta`：

1. 用户消息记录 `model_config_id`
2. 助手消息冗余记录：
   - `model_config_id`
   - `model_name`
   - `display_name`
   - `base_url`

这样即使管理员后续修改模型配置名称，历史消息仍然能保留当时的调用上下文。

## API 设计

### 鉴权相关接口补充

为了让前端识别管理员身份，需要在当前认证返回中增加 `is_admin` 字段：

1. `POST /api/auth/login`
2. `POST /api/auth/register`
3. `GET /api/auth/me`

返回中的 `user` 保持不变，也可以直接在顶层响应增加 `is_admin`。只要前后端结构统一即可。

### 管理端模型配置接口

新增管理员路由组，例如：

1. `GET /api/admin/model-configs`
   返回全部模型配置。
2. `POST /api/admin/model-configs`
   新建模型配置。
3. `PATCH /api/admin/model-configs/{id}`
   编辑模型配置。
4. `DELETE /api/admin/model-configs/{id}`
   删除模型配置。

这些接口都要求当前用户命中 `ADMIN_EMAILS`。

### 前台可用模型接口

新增普通登录用户可访问接口：

1. `GET /api/model-configs/enabled`

返回字段建议只包含工作台必要信息：

1. `id`
2. `name`
3. `model`

不返回 `api_key` 和 `base_url`，避免前台暴露后台配置细节。

### 对话发送接口扩展

现有接口：

1. `POST /api/sessions/{session_id}/messages`

请求体从：

```json
{ "content": "..." }
```

扩展为：

```json
{ "content": "...", "model_config_id": "..." }
```

后端校验规则：

1. `model_config_id` 必填。
2. 对应模型配置必须存在。
3. 对应模型必须为启用状态。
4. 当前用户必须拥有目标会话。

## Web 交互设计

### 管理页

新增页面路由：

1. `/admin/models`

页面结构采用两段式：

1. 上方为新建 / 编辑表单
2. 下方为模型列表

表单字段：

1. `name`
2. `base_url`
3. `model`
4. `api_key`
5. `enabled`

交互规则：

1. 点击“编辑”时回填表单。
2. 点击“新建”时清空表单。
3. 删除前二次确认。
4. 保存成功后刷新列表。
5. 非管理员访问时直接拒绝访问并返回工作台或 403 页面。

### 工作台模型选择

模型选择器放在 `Composer` 区域，与消息发送行为靠近。

原因：

1. 它是本次发送参数，不是整个会话属性。
2. 用户更容易理解“这条消息会使用哪个模型”。

交互规则：

1. 进入工作台时加载已启用模型列表。
2. 默认选中前端记忆的上一次选择。
3. 用户可以在每次发送前切换模型。
4. 发送时将当前 `model_config_id` 一并提交给后端。
5. 如果当前已没有任何启用模型：
   - 禁用发送按钮
   - 提示“当前暂无可用模型，请联系管理员配置”
6. 如果用户当前选中的模型被禁用并重新拉取列表后不存在：
   - 自动回退到第一项可用模型
   - 若无可用模型则进入禁用态

### 前台展示信息

工作台只展示：

1. 模型展示名 `name`
2. 可选展示真实模型名 `model`

不展示：

1. `api_key`
2. `base_url`

## 后端调用设计

### 调用流程

消息服务处理流程调整为：

```text
Workspace Composer
  -> POST /api/sessions/{session_id}/messages
  -> validate model_config_id
  -> load conversation context
  -> call OpenAI-compatible API
  -> persist user/assistant messages with model meta
  -> return SSE events
```

### OpenAI 兼容适配层

新增一个独立服务层，例如：

1. `app/services/model_gateway.py`

职责：

1. 根据模型配置组装请求地址与请求头
2. 以 OpenAI 兼容协议发起聊天请求
3. 解析响应文本
4. 将上游错误转为统一异常

建议输入：

```python
generate_reply(
    base_url: str,
    api_key: str,
    model: str,
    messages: list[dict[str, str]],
) -> str
```

工作方式：

1. 从当前会话历史拼装 OpenAI 兼容消息数组
2. 增加系统提示词，约束返回风格与当前产品定位
3. 调用 `POST {base_url}/chat/completions` 或兼容路径
4. 从响应中提取助手回复文本

### 与现有 Agent Runtime 的关系

当前 `run_agent(state, user_input)` 返回的是固定回复逻辑。

本次建议分两层处理：

1. 短期保留“动作决策”结构，用于 `action.decided`
2. 将回复生成替换为外部模型调用

可行路径：

1. 继续由当前运行时负责生成下一步动作
2. 将动作、状态摘要、历史消息拼进 prompt
3. 由外部模型生成真正的自然语言回复

这样可以避免一次性推翻当前状态驱动结构。

## 错误处理

### 管理页校验

保存模型时做最小必要校验：

1. `name` 必填
2. `base_url` 必填且必须是合法 URL
3. `model` 必填
4. `api_key` 必填

### 对话发送错误

后端错误处理策略：

1. 模型不存在：返回 404
2. 模型已禁用：返回 400 或 409
3. 模型配置缺失关键字段：返回 400
4. 上游鉴权失败：返回清晰错误
5. 上游超时或 5xx：返回“模型调用失败，请稍后重试”
6. 上游响应格式不兼容：返回“模型响应格式不受支持”

### 前端错误展示

1. 继续复用当前 toast + composer 错误提示
2. 不做静默降级
3. 不自动切换到其他模型

原因：

1. 自动回退会让用户误以为本次使用的是所选模型
2. 审计和问题排查会失真

## 安全与风险

本次设计存在明确风险，需要在实现中写清楚：

1. `api_key` 明文存储、明文回显，存在高风险。
2. `ADMIN_EMAILS` 适合当前阶段，但不适合复杂权限体系。
3. `base_url` 为外部地址，后端需要限制请求超时并做好异常兜底。

## 测试策略

### 后端测试

需要覆盖：

1. `ADMIN_EMAILS` 白名单判定
2. 管理接口的管理员访问控制
3. 模型配置 CRUD
4. 已启用模型列表过滤
5. 发送消息时 `model_config_id` 校验
6. OpenAI 兼容接口调用成功分支
7. OpenAI 兼容接口调用失败分支
8. 消息 `meta` 中记录模型元数据

### 前端测试

需要覆盖：

1. 管理页加载列表
2. 创建模型
3. 编辑模型
4. 删除模型
5. 非管理员不可访问管理页
6. 工作台模型下拉加载
7. 发送消息时携带所选 `model_config_id`
8. 无可用模型时发送按钮禁用

### 集成验证

至少验证以下闭环：

1. 管理员新增并启用一个模型
2. 普通用户进入工作台能看到该模型
3. 普通用户选择该模型发送消息
4. 后端确实按该 `base_url + api_key + model` 发起请求
5. 历史消息中保留模型元数据

## 验收标准

满足以下条件可视为本次设计落地成功：

1. 管理员可以通过页面管理平台模型配置。
2. 普通用户只能看到已启用模型。
3. 工作台支持按消息切换模型。
4. 对话发送接口必须携带模型配置 ID。
5. 后端通过 OpenAI 兼容接口发起真实请求。
6. 历史消息可以追溯本次使用的模型。
7. 非管理员无法访问管理能力。
